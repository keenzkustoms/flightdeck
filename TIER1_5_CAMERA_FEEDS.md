# Tier 1.5 — Live camera feeds in hover popover

## Context

The hover popover currently shows static slicer thumbnails. This spec upgrades it to live camera feeds when a print is active. The popover API already accepts an `image_type` field (`"static" | "mjpeg" | "webrtc"`) — this spec is about wiring the live streams to it.

Three different camera setups, three different transport stories. All converge on the same frontend component.

## What changes

When a card is in `PRINTING` (or `PAUSED`, or `FINISHED` while cooling) and the user hovers, the popover shows a live camera frame instead of (or above) the static thumbnail. When the printer is `IDLE` or `OFFLINE`, no popover appears at all — same as current behaviour.

If the live stream fails to load within a short timeout (~2s), fall back to the static slicer thumbnail. If neither is available, show the existing metadata-only fallback. Three-tier graceful degradation.

## Per-printer setup

### Voron Greyhound — Logitech C270

The C270 is connected to the Voron's CB2 host (the box running Klipper + Moonraker, not the flightdeck Pi). Confirm one of the following is running on that host:

- **`crowsnest`** (most likely if you've used MainsailOS) — exposes `/webcam/?action=stream` for MJPEG and `/webcam/?action=snapshot` for stills
- **`mjpg-streamer`** directly — exposes `http://<host>:8080/?action=stream`

Stream URL goes in `printers.yaml` per-printer:

```yaml
- id: greyhound
  # ... existing fields ...
  camera:
    type: "mjpeg_direct"
    stream_url: "http://192.168.x.x/webcam/?action=stream"
    snapshot_url: "http://192.168.x.x/webcam/?action=snapshot"
```

The browser fetches the MJPEG stream directly — no proxy needed. Simplest case.

### Bambu X1C and H2D — built-in cameras

Both printers expose an RTSP stream over LAN at `rtsps://bblp:<access_code>@<printer_ip>:322/streaming/live/1`. The `bblp` username is fixed; the password is the printer's 8-digit access code.

Browsers can't render RTSP directly. Need a proxy. Two options, in order of recommendation:

**Option A: `bambulabs-api` built-in helper (if available)** — check whether the library exposes a camera streaming method. If yes, use it.

**Option B: ffmpeg sidecar process** — spawn one ffmpeg process per active Bambu camera, pulling RTSP and re-encoding to MJPEG served on a local port:

```bash
ffmpeg -rtsp_transport tcp \
       -i "rtsps://bblp:ACCESS_CODE@192.168.x.x:322/streaming/live/1" \
       -f mpjpeg -q:v 5 -r 10 \
       http://127.0.0.1:8090/x1c.mjpg
```

Run via Python's `asyncio.create_subprocess_exec` with proper cleanup on shutdown. Start lazily — only spawn when first requested, idle out after ~60s of no requests to save CPU. Don't keep RTSP connections open permanently; the Pi 5 has the headroom but Bambu printers can get cranky with persistent camera connections.

Config:

```yaml
- id: x1c
  # ... existing fields ...
  camera:
    type: "bambu_rtsp"
    # stream_url is generated by the proxy, not configured directly
    # the proxy reads serial + access_code from the connection config
```

Proxy endpoint pattern: `http://flightdeck:8000/api/camera/{printer_id}/stream`. Frontend hits this URL; backend handles RTSP→MJPEG translation.

## Frontend integration

The popover component already has the `image_type` branch from `TIER1_HOVER_THUMBNAIL.md`. Extend it:

```jsx
function PreviewMedia({ image_type, image_url, fallback_thumbnail_url, ... }) {
  const [streamFailed, setStreamFailed] = useState(false);
  
  if (image_type === "mjpeg" && !streamFailed) {
    return (
      <img
        src={image_url}
        onError={() => setStreamFailed(true)}
        // browsers will show MJPEG streams as auto-refreshing images
      />
    );
  }
  
  // fallback to static thumbnail
  if (fallback_thumbnail_url) {
    return <img src={fallback_thumbnail_url} />;
  }
  
  // fallback to placeholder + metadata only (existing behaviour)
  return <PlaceholderIcon />;
}
```

Backend `/api/printer/{id}/preview` returns both fields so the frontend can degrade gracefully without a second request:

```python
class PrintPreview(BaseModel):
    image_url: str | None              # live stream URL when available
    image_type: Literal["static", "mjpeg", "webrtc"]
    fallback_thumbnail_url: str | None # static slicer thumb, always populated if exists
    # ... existing metadata fields ...
```

## State-aware behaviour

| Printer state | Popover content |
|---|---|
| `IDLE` | No popover (existing behaviour) |
| `OFFLINE` | No popover (existing behaviour) |
| `PRINTING` | Live camera if available, else static thumbnail, else placeholder |
| `PAUSED` | Live camera (still useful — show what's frozen on the bed) |
| `FINISHED` (within 30min window, hotend > 50°C) | Live camera — useful to see the finished print before plate removal |
| `FINISHED` (cooled) | Static thumbnail of what was printed |
| `ERROR` | Live camera (the most diagnostically useful thing in an error state) |

## Performance and resource notes

- **Lazy-start ffmpeg processes.** Only spawn when the popover is actually requested. Tear down after 60s of inactivity.
- **One ffmpeg per Bambu printer max.** If multiple users hover the same card simultaneously, they share the proxy stream — don't spawn parallel processes.
- **Cap stream FPS at 10.** The slicer thumbnail you replaced was a static image; 10fps is plenty to feel "live" in a hover popover and keeps CPU sane.
- **MJPEG is a stream of full JPEG frames** — bandwidth-heavy compared to H.264, but trivially browser-compatible. WebRTC would be nicer for the Bambus but is a larger build; not worth it for v1.
- **No persistent connections to printer cameras.** RTSP stays open only while ffmpeg is running, which is only while someone's hovering. Don't try to "warm up" streams in the background.

## Out of scope

- Embedding camera feeds permanently on Tier 1 cards (resist — kills the glance hierarchy)
- Recording video to disk
- Time-lapses
- Camera controls (pan/tilt/zoom — N/A for these cameras anyway)
- Multi-camera views per printer
- WebRTC (later, when latency matters more than implementation complexity)
- Authenticating the proxy endpoint (LAN-only deployment, no auth needed for v1)

## Implementation order

1. **Voron MJPEG first** — simplest path, exercises the popover branching logic with a real live stream. Confirms the frontend component handles streaming correctly before adding proxy complexity.
2. **Bambu RTSP proxy** — pick `bambulabs-api` helper if it exists, else ffmpeg subprocess. Start with one Bambu (X1C, since it's already printing).
3. **Apply to second Bambu** — same proxy pattern, second config entry.
4. **State-aware behaviour table** — wire the popover to consult printer state when deciding what to show.
5. **Graceful degradation** — wire the fallback chain (live → static → placeholder).

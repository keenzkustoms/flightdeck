What it does

Stop checking five screens to know what the room is doing.

Fleet Wall

See live cameras and printer state across the whole room without opening Bambu Studio, Mainsail, and browser tabs.

Printer cockpit

Keep the camera, live controls, temperatures, print state, filament route, and AMS loadout together on one page.

AMS Doctor

Compare what the printer reports against what Flightdeck knows, then deliberately trust the printer or Flightdeck.

Spool tracking

Track real rolls by shelf, AMS slot, weight, colour, brand, profile, label, humidity, and low-stock warnings.

Queue and Print Bay

Browse the Print Vault, queue jobs, move files, and prepare work without losing the printer context.

Orca handoff

Open source models in managed Orca, apply printer profiles, slice to the vault, then queue or send the result.

Diagnostics

Download support logs, check setup health, spot camera pressure, and make tester feedback easier to understand.

Phone and bench ready

Check alerts, cameras, jobs, and spool state from the machine, the desk, or your phone over your own network.

The difference

Inventory apps track stock. Flightdeck runs the room.

Spools matter, but they are only one part of print-farm life. Flightdeck brings printers, cameras, AMS, queue, slicer handoff, failures, moisture watch, maintenance, and support diagnostics into the same operating surface.

Built from a real bench

Flightdeck is being shaped by daily testing on Bambu H2D, Bambu X1C, Voron 2.4, and expanding tester fleets.

Live view

The printer page is the cockpit.

Flightdeck keeps the camera as the hero, then wraps it with the details that matter during a print: current job, action buttons, temperatures, loaded filament, AMS humidity, and live filament routing.

Why it exists

Mixed fleets should not feel like three separate jobs.

Flightdeck started from a real bench: a Bambu H2D, a Bambu X1C, and a Voron 2.4. It is built around the way printers are actually used: cameras, queues, filament, failures, maintenance, and quick decisions from the bench or a phone. The goal is not to replace every slicer or printer UI. The goal is to reduce the daily context switching and make the important signals harder to miss.

Current focus

Flightdeck is moving from private homelab build toward open-source feedback. It is already proven on Bambu and Voron, and now needs testers with other printer ecosystems.

Install

Run it on a Pi or Windows machine. Keep your data local.

# Pi install
curl -fsSL https://raw.githubusercontent.com/Kidabah/flightdeck/main/scripts/install-pi.sh | bash

# Windows install
powershell -ExecutionPolicy Bypass -File .\scripts\windows\install-windows.ps1

Easy as 1-2-3

  1. Install Flightdeck on the Pi.
  2. Add printers from Settings.
  3. Add spools from the Spools screen.

Pi 5 4 GB for small fleets, Pi 5 8 GB as the recommended default, and Pi 5 16 GB for 10+ printers or camera-heavy rooms. Pi 4 can run light installs, but Pi 5 is the main target.

Optional hardware: Dymo USB scale and Brother QL-700 label printer.

Tester path

First run should feel safe, not technical.

Flightdeck ships with Demo Mode, a standalone `/demo` page after install, and a Flight Manual so a new tester can explore the dashboard, cameras, printer pages, Print Bay, spools, failures, telemetry, and maintenance before touching real printer controls.

Best first pass

Install on a Pi or Windows machine, open `/demo` or Demo Mode, confirm Setup Health, add one printer, then browse read-only screens before testing queue, file, AMS, or hardware actions.
